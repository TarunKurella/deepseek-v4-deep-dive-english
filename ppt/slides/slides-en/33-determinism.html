MODULE 03 · BATCH-INVARIANT & DETERMINISTIC

Bit-exact alignment between training and inference

No matter where the same token appears inside a batch, the output bits stay identical. That means loss spikes can be debugged directly against training traces, and production anomalies can be reproduced faithfully in the training environment.
DIFFICULTY 01
Attention
Split-KV introduces wave quantization, so the accumulation order for the same token changes with batch placement and the output drifts.
Dual-kernel strategy: one kernel runs the full sequence on a single SM for throughput, while another uses multiple SMs only for the last partially filled wave to cut latency. The accumulation order is kept strictly aligned.
DIFFICULTY 02
Matmul
cuBLAS cannot make results batch-invariant. But giving up split-k entirely would normally cost performance.
Swap the engine: cuBLAS is dropped in favor of DeepGEMM, with enough targeted optimizations to recover most of the performance sacrificed by abandoning split-k.
DIFFICULTY 03
Backward pass
The main offender is atomicAdd. Floating-point addition is not associative, so different execution orders produce different results.
Independent buffers + globally ordered reduction: each SM accumulates into its own buffer first, then a deterministic global reduction happens afterward. MoE also preprocesses token order.
The real significance of these three fixes together is that every token in RL rollout can match the training environment bit for bit. For any team doing large-model post-training and RL, that is an absurdly valuable base capability.
